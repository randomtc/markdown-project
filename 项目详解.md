# Markdown 渲染器项目详解

## 项目概述

这是一个基于 React + TypeScript + Vite 的 Markdown 编译器/渲染器，支持流式渲染、自定义组件配置和丰富的交互功能。

## 核心架构

### 三层渲染管道

```
Markdown 文本 → 流式预处理 → HTML → React 元素
```

| 层级          | 文件              | 职责                     |
| ------------- | ----------------- | ------------------------ |
| 流式预处理    | `useStreaming.ts` | 处理流式输入，生成占位符 |
| Markdown 解析 | `Parser.ts`       | Marked 解析为 HTML       |
| React 渲染    | `Renderer.ts`     | HTML 转换为 React 元素   |

## 核心模块详解

### 1. useStreaming - 流式渲染 Hook

**文件**: `src/hooks/useStreaming.ts`

#### 核心设计：Streaming Tokenizer 模式

```
输入字符流 → Token识别 → 状态缓存 → 占位符生成
```

#### 关键优化：增量处理

```typescript
// 只处理新增的字符，避免重复计算
const chunk = text.slice(currentCache.processedLength);
```

**性能对比**:

- 全量处理: O(n²) - 1000字符需要处理 1000+999+998+... 次
- 增量处理: O(n) - 1000字符只处理 1000 次

#### 状态流转

```
Text --"!"--> Image(阶段1) --"](..."--> Image(阶段2) --")"--> Text
```

#### 缓存结构

```typescript
interface StreamCache {
  pending: string; // 当前未完成的 token 内容
  completeMarkdown: string; // 已确认完成的 Markdown
  processedLength: number; // 已处理的长度
  token: StreamTokenType; // 当前 token 类型
}
```

#### 正则识别规则

```typescript
const STREAM_INCOMPLETE_REGEX = {
  image: [
    /^!\[[^\]\r\n]{0,1000}$/, // 阶段1: ![ 后无 ]
    /^!\[[^\r\n]{0,1000}\]\(*[^)\r\n]{0,1000}$/, // 阶段2: ]( 后无 )
  ],
  link: [
    /^\[[^\]\r\n]{0,1000}$/, // 阶段1: [ 后无 ]
    /^\[[^\r\n]{0,1000}\]\(*[^)\r\n]{0,1000}$/, // 阶段2: ]( 后无 )
  ],
};
```

### 2. Parser - Markdown 解析器

**文件**: `src/core/Parser.ts`

#### 职责

- 使用 Marked 库解析 Markdown 为 HTML
- 支持自定义渲染器（链接、图片、代码块等）
- 流式状态检测：为标签添加 `data-stream-status` 属性

#### 关键特性

**流式状态标记**:

```typescript
// 图片渲染时检测完整性
const isComplete = completeImageRegex.test(raw);
const streamStatus = isComplete ? "done" : "loading";
return `<img ... data-stream-status="${streamStatus}" />`;
```

**XSS 防护**:

```typescript
const escapeHtml = (html: string): string => {
  // 转义 &, <, >, ", ', / 等特殊字符
};
```

### 3. Renderer - React 渲染器

**文件**: `src/core/Renderer.ts`

#### 职责

- 使用 `html-react-parser` 将 HTML 转换为 React 元素
- 支持自定义组件映射
- XSS 防护：DOMPurify 净化

#### 组件映射机制

```typescript
const components = {
    img: SkeletonImage,      // 自定义图片组件
    a: CustomLink,           // 自定义链接组件
    code: CodeBlock,         // 自定义代码块
    'incomplete-image': () => <span>图片加载中...</span>,
};
```

### 4. Markdown 组件

**文件**: `src/components/Markdown.tsx`

#### 核心流程

```typescript
export const Markdown: React.FC<MarkdownProps> = ({
    content,
    streaming,
    components,
}) => {
    // 1. 流式预处理
    const streamingContent = useStreaming(content, streaming);

    // 2. 解析并渲染（带缓存）
    const reactContent = useMemo(() => {
        const html = parseMarkdown(streamingContent);
        return renderHtml(html, { components });
    }, [streamingContent, components]);

    return <div>{reactContent}</div>;
};
```

#### useMemo 优化

缓存 `Markdown → HTML → React` 的转换结果，避免配置切换时重复解析相同内容。

**依赖项设计**:

```typescript
[streamingContent, components, openLinksInNewTab];
```

- `streamingContent` 变化 → 必须重新解析（新内容）
- `components` 变化 → 必须重新渲染（组件映射变了）

## 流式渲染流程详解

### 完整流程图

```
用户输入/AI流式输出
    ↓
[useStreaming] 逐字符处理
    ↓
检测 Token 类型 (Image/Link/Text)
    ↓
更新 StreamCache (pending/complete)
    ↓
生成占位符 (incomplete-image/link)
    ↓
[Parser] Marked 解析为 HTML
    ↓
添加 data-stream-status 属性
    ↓
[Renderer] 转换为 React 元素
    ↓
自定义组件映射
    ↓
渲染到页面
```

### 占位符机制

**问题**: 流式输出时 `![图片` 还未闭合，直接解析会报错或渲染异常。

**解决方案**:

```typescript
// 未闭合时生成占位符
const placeholder = `<incomplete-image data-raw="![图片" />`;

// 注册占位组件
const components = {
    'incomplete-image': (props) => <SkeletonImage loading />,
};

// 闭合后正常渲染
// `![图片](url)` → `<img src="url" />`
```

## 性能优化点

### 1. 增量处理

- 只处理新增字符，避免 O(n²) 复杂度

### 2. useMemo 缓存

- 缓存解析结果，配置切换时不重复解析

### 3. useRef 缓存

- StreamCache 使用 useRef，避免每次渲染重建

### 4. 防抖渲染

- Mermaid 等复杂组件使用防抖，避免频繁渲染

## 扩展指南

### 添加新的 Token 类型

**步骤**:

1. `StreamTokenType` 枚举添加新类型
2. `STREAM_INCOMPLETE_REGEX` 添加正则
3. `tokenRecognizerMap` 添加识别器
4. 注册占位组件

**示例**: 添加表格支持

```typescript
// 1. 添加枚举
enum StreamTokenType {
  Table = "table",
}

// 2. 添加正则
const STREAM_INCOMPLETE_REGEX = {
  table: [/^\|[^\n]*$/, /^\|[^\n]*\n\|[-:\| ]+$/],
};

// 3. 添加识别器
tokenRecognizerMap[StreamTokenType.Table] = {
  tokenType: StreamTokenType.Table,
  isStartOfToken: (s) => s.startsWith("|"),
  isStreamingValid: (s) =>
    STREAM_INCOMPLETE_REGEX.table.some((re) => re.test(s)),
};
```

## 面试要点

### 1. 架构设计

- 三层管道设计，职责分离
- 纯函数实现，可预测性

### 2. 性能优化

- 增量处理从 O(n²) 到 O(n)
- useMemo/useRef 减少重复计算

### 3. 扩展性

- Token 识别器模式，易于扩展
- 组件映射机制，灵活定制

### 4. 边界处理

- 代码块内跳过识别
- URI 编码安全处理
- 输入不连续时重置缓存

### 一句话总结

> "用状态机解决流式语法解析问题，用缓存优化性能，用占位符提升体验，设计上预留了扩展点。"
